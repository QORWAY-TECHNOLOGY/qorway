# System Map

**QORWAY Decision Intelligence Infrastructure**

> From fragmented enterprise systems  
> → to causal, constrained, executable intelligence.

---

## 1. Definition

The QORWAY System Map defines how the full infrastructure operates as a unified decision intelligence runtime.

It shows how data becomes reasoning, reasoning becomes governed decisions, decisions become execution, and execution becomes feedback.

QORWAY is not a static architecture.

It is a closed-loop system.

---

## 2. System Purpose

QORWAY exists to transform enterprise decision-making into a structured, auditable, and executable system.

It connects:

- enterprise data
- Knowledge Graphs
- causal reasoning
- Domain Packs
- constraint validation
- event orchestration
- execution optimization
- agent execution
- feedback learning

The goal is to make decisions:

- causal
- contextual
- constrained
- executable
- traceable
- self-improving

## 4. Five Core Layers

QORWAY operates as a five-layer decision infrastructure:

```text
1. Atlas         → Causal Intelligence Engine
2. Domain Packs  → Context & Business Intelligence Layer
3. PolicyCore    → Constraint Validation Layer
4. PulseFlow     → Execution Orchestration Layer
5. GreenCore     → Execution Optimization Layer
```

Each layer has a distinct responsibility.

No layer replaces another.

---

## 5. Layer Responsibilities

| Layer        | Function                | Question Answered                       |
| ------------ | ----------------------- | --------------------------------------- |
| Atlas        | Causal reasoning        | What should happen?                     |
| Domain Packs | Domain context          | How does this domain work structurally? |
| PolicyCore   | Constraint validation   | Is this allowed to exist?               |
| PulseFlow    | Execution orchestration | How does this move through systems?     |
| GreenCore    | Execution optimization  | Where and how should this execute?      |

## 6. Supporting System Layers

In addition to the five core layers, QORWAY relies on supporting infrastructure layers.

---

### 6.1 Knowledge Graph Layer

The Knowledge Graph is the memory and causal structure layer of QORWAY.

It stores:

* entities
* indicators
* actions
* risks
* gates
* outcomes
* decisions
* causal relationships
* execution feedback

It exists in two forms:

```text
Universal Knowledge Graph
Private Tenant Knowledge Graph
```

The Universal Knowledge Graph contains reusable, non-identifiable causal structures.

The Private Tenant Knowledge Graph contains tenant-specific operating reality.

---

### 6.2 Agent Runtime Layer

The Agent Runtime executes bounded tasks generated by validated decisions.

Agents may perform:

* monitoring
* validation
* reporting
* workflow execution
* system interaction
* feedback collection

Agents do not reason independently from Atlas.

Agents do not execute without PulseFlow.

Agents remain bounded by PolicyCore and tenant policies.

---

### 6.3 Governance Layer

Governance is embedded across the full runtime.

It includes:

* causal evidence logging
* PolicyCore constraint validation
* tenant boundaries
* human approval where required
* execution traceability
* audit-ready logs

Governance is not post-hoc.

It is part of the system flow.

---

### 6.4 Security Layer

Security is enforced across all components.

Core principles:

* Zero Trust
* strict tenant isolation
* encrypted data at rest
* encrypted data in transit
* event-level access control
* audit logs
* no cross-tenant leakage

## 12. Separation of Concerns

QORWAY enforces strict separation between layers.

| Layer        | Can Do                | Must Not Do          |
| ------------ | --------------------- | -------------------- |
| Atlas        | Reason causally       | Execute directly     |
| Domain Packs | Provide context       | Override constraints |
| PolicyCore   | Validate constraints  | Generate decisions   |
| PulseFlow    | Orchestrate execution | Reason causally      |
| GreenCore    | Optimize execution    | Override PolicyCore  |
| Agents       | Execute bounded tasks | Bypass PulseFlow     |

This separation protects auditability, scalability, and regulatory alignment.

---

## 13. PolicyCore Position

PolicyCore is the constraint authority.

It decides whether a decision is:

```text
APPROVED
MODIFIED
REJECTED
```

No execution can occur unless PolicyCore has validated the decision.

Core rule:

> No decision becomes executable until PolicyCore validates it.

---

## 14. GreenCore Position

GreenCore is the execution optimization authority.

It decides how validated execution should happen across:

```text
local
edge
cloud
```

It optimizes for:

* cost
* latency
* carbon footprint
* data sovereignty
* tenant execution policy

Core rule:

> GreenCore optimizes only paths that PolicyCore has approved.

---

## 15. PulseFlow Position

PulseFlow is the execution nervous system of QORWAY.

It transforms validated decisions into:

* events
* workflows
* agent tasks
* execution logs
* feedback signals

Core rule:

> No execution should happen outside PulseFlow.

---

## 16. Domain Pack Position

Domain Packs are the context intelligence layer.

They define:

* graph structure
* causal rules
* decision policies
* events
* agents
* feedback rules
* systemic impact model

Core rule:

> Domain Packs structure domain reality. They do not override PolicyCore.

---

## 17. Feedback Model

Feedback is captured after execution and reinjected into the Knowledge Graph.

Feedback may include:

* expected vs actual outcome
* delay
* cost
* risk change
* compliance result
* carbon impact
* execution performance
* human approval result

This allows future decisions to improve.

---

## 18. Auditability Model

Every system action must produce traceable evidence.

Audit artifacts may include:

* decision trace
* causal chain
* Domain Pack version
* PolicyCore result
* PulseFlow event log
* GreenCore routing trace
* execution log
* feedback record

A system state is invalid if it cannot be audited.

---

## 19. Multi-Tenant System Boundary

Each tenant has isolated:

* data
* Knowledge Graph
* Domain Pack instantiation
* PolicyCore constraints
* PulseFlow execution events
* GreenCore execution policies
* audit logs
* feedback loop

No tenant can access or infer another tenant’s private graph or execution state.
